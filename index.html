Enhancement 1: Software Design and Engineering

Artifact Overview

The artifact I chose for enhancement is a mobile inventory management application I created during my CS 360: Mobile Architecture and Programming course last term. The application lets users log in, view, and manage inventory items, and conduct typical CRUD (Create, Read, Update, Delete) activities on a local SQLite database. The initial version of the program was functional, but it lacked architectural structure and had overlapping between user interface code, logic, and data handling.

Reason for Selection & Improvements

I chose this item for enhancement because it allowed me to demonstrate a stronger grasp of software engineering design concepts, specifically the Model View Controller (MVC) architectural pattern. The app was more difficult to maintain, expand, and test in the previous codebase since a large portion of the logic, such as managing user actions and data manipulation, was tightly interconnected with the UI Activities.

To fix this, I changed the project's structure to follow the MVC pattern, where the program is now divided into three separate layers (model, view, and controller).

  • Model: This handles the data logic and SQLite database operations.
  • View: This consists of the XML layout files (which remain in the res/layout directory as required by Android) and activity classes responsible for displaying data and handling user input.
  • Controller: This is a layer with a new specialized InventoryController class that handles business logic and serves as a link between the model and the view layers.

This restructure encourages separation of responsibilities, enhances code modularity, and promotes scalability and maintainability. I also restructured the current Kotlin files into appropriate packages (controller, view, and model) to match the architectural change and make the codebase simpler and more understandable to navigate.

Course Outcomes Alignment

This enhancement aligns with the following course outcomes:

  • Design and evaluate computing solutions that solve a given problem using algorithmic principles and computer science practices and standards appropriate to its solution while managing the trade-offs involved in design choices.
  • Demonstrate an ability to use well-founded and innovative techniques, skills, and tools in computing practices for the purpose of implementing computer solutions that deliver value and accomplish industry-specific goals.

By rebuilding this artifact with the MVC pattern, I proved my ability to use core computer science design concepts to improve an existing application. I made deliberate design trade-offs to strike a compromise between short-term goals like retaining functionality with SQLite and long-term scalability like building a structure suited for Firebase (a future enhancement). I also demonstrated my ability to design practical solutions with clean, testable, and maintainable code structures by utilizing modern development tools and techniques.

Reflection on the Enhancement Process

The process of improving the application was both tough and very worthwhile. I began by noting the flaws in the previous structure, specifically how roles were not clearly split. The activity classes were doing too much by handling both the UI and the application logic, making the app difficult to comprehend and practically impossible to unit test.

To address this, I added a new InventoryController class that handles interactions between the user interface and the data model. I also refactored existing functionality to comply with MVC standards, which included rewriting various sections of code to make sure that each layer was well defined and connected with others in a clean/organized manner. I also modified the view layer to delegate logical operations to the controller, and I packed the SQLite operations in the model layer.

One of the most difficult aspects of this development for me was rearranging the codebase while avoiding bugs. Since Android apps are tightly coupled between UI and functionality, getting rid of this dependency to make sure I had a clean design required a lot of meticulous planning. I needed to make sure that each layer communicated effectively (especially between the controller and the model) without breaking encapsulation.

My understanding of architectural design patterns in mobile development has grown as a result of this enhancement. I discovered that even a small project could benefit tremendously from following MVC principles. It also reaffirmed how important modular code is for maintainability and for making future improvements, like integrating Firebase later on, much simpler. Overall, by utilizing industry standard best practices, I definitely obtained practical skills in refactoring code, restructuring Kotlin files based on their purposes, and enhancing a mobile application's overall structure.

Enhancement 2: Algorithms and Data Structures

Artifact Overview

The artifact is an Android inventory management application I developed initially during my CS 360: Mobile Architecture and Programming course last term. The app allows users to add, view, update, and delete inventory items stored locally in an SQLite database. The program accurately recorded basic item details such as name, amount, and category, but it did not enable regulating item expiry dates, which is an important function for inventory control in many real-world scenarios. This addition, implemented in this enhancement, included expiry date tracking features to help with inventory monitoring and management.

Reason for Selection & Improvements

I chose this item for my ePortfolio because it clearly illustrates my ability to enhance an existing program by incorporating new data structures and introducing algorithmic logic to improve its functionality and usability. To enhance the artifact, I started by modifying the data model in Item.kt to include a new property called expirationDate, which is represented as a string in "yyyy-mm-dd" format. This necessitated changes to the SQLite database schema in DatabaseHelper.kt, where I added an expiration_date field to the inventory_items table and increased the database version to accommodate schema upgrades. I also reviewed all important CRUD operations (insertInventoryItem, getAllInventoryItems, and updateInventoryItem) to ensure that expiry data is correctly saved, retrieved, and synced with the database.

Beyond the data layer, I improved the user experience by altering many XML layout files. I added an EditText to dialog_add_item.xml and dialog_item_details.xml so that users could reliably insert or alter expiration dates. I also added a new TextView to inventory_item_row.xml to display each item's expiration date in the inventory list for an easy visual reference. Moreover, I changed the main inventory layout/dashboard, activity_inventory.xml, to include an expiration column heading, ensuring clear alignment and readability.

For better usability, I also added dynamic filtering and sorting algorithms to InventoryAdapter.kt, allowing users to organize inventory items in real time based on stock status and expiry dates. I added a filter spinner and a sort button to a new control bar in activity_inventory.xml and connected them in InventoryActivity.kt with listeners that interact with the adapter's logic. This connection enables the app to respond to user selections in real time, enhancing navigation and inventory management.

These enhancements demonstrate my ability to manage changing data structures, apply algorithmic concepts to data manipulation, and synchronize data flow across the database, adapter, and user interface. I believe the artifact was significantly improved by introducing expiration tracking, increasing data visibility, and enabling responsive, user-driven sorting and filtering capabilities that will facilitate advanced inventory management functions in the future like emphasizing low stock items or making alerts for products that are about to expire.

This enhancement aligns with the following course outcomes:

Enhancement 3: Databases

Artifact Overview

The artifact is an Android inventory management application I developed initially during my CS 360: Mobile Architecture and Programming course last term. The app allows users to add, view, update, and delete inventory items stored locally in an SQLite database. In this enhancement, I replaced the local database with Firebase for user authentication and real-time cloud inventory management. This change enables a more scalable and modern application architecture.

Reason for Selection & Improvements

I chose this item for my ePortfolio because it illustrates my abilities to improve an application I already created using cloud based tools. Firebase integration is a significant update which required me to become familiar with asynchronous operations, real time data syncing, and user authentication (these are all essential skill sets in today's mobile development scene which is why I thought it was perfect to showcase in my ePortfolio).

This enhancement required that I changed many files in the application, specifically:

  • build.gradle (both at the app and project level): I added the necessary Firebase dependencies, including firebase-auth and firebase-database, and set up the google-services plugin to make sure Firebase was properly initialized. These adjustments were necessary to provide cloud support in the app.
  • FirebaseUtil.kt (new file I created): I designed a centralized utility class that handles all Firebase activities. This contains user registration, login, and CRUD actions on inventory items. I also made sure that each item stored in Firebase has its own unique ID, which can then be used to quickly track and update data.
  • Item.kt: I added an id field to the data model with a default value of null. I additionally provided default values for other properties that facilitate Firebase's object mapping and deserialization. These modifications were required to be able to access and modify inventory data stored in Firebase.
  • InventoryActivity.kt: Here I replaced all SQLite references with calls to the Firebase utility. This included changing the way data was retrieved, displayed, and updated in the app. I also handled asynchronous Firebase data retrieval with callbacks.
  • InventoryAdapter.kt: I changed the adapter to use Firebase's id attribute for delete and update operations. This update guaranteed that the right inventory item was changed in the cloud database and that the UI reflects these real time changes.
  • LoginActivity.kt: I changed the login logic to utilize Firebase authentication. This not only allowed users to register and log in using their email addresses and passwords, but it also increased security and scalability. The login screen now uses FirebaseAuth to authenticate users before providing access to the inventory dashboard.

These modifications demonstrate my abilities to provide secure authentication, handle cloud-based data, and redesign an application for more scale and easy maintenance.

Course Outcomes Alignment

This enhancement aligns with the following course outcomes:

  • Demonstrate an ability to use well-founded and innovative techniques, skills, and tools in computing practices for the purpose of implementing computer solutions that deliver value and accomplish industry-specific goals.

By integrating Firebase, I was able to tackle common software concerns including safe user authentication and real time data storage. I believe this enhancement added significant value to the application by allowing for multi-device access, user-specific data management, and a more flexible architecture. Using Firebase also illustrates my proficiency with modern frameworks and industry technologies, as it reflects the type of cloud-based solutions frequently used in corporate mobile apps.

Reflection on the Enhancement Process

When I reflect on the process of enhancing this artifact, I find that improving this artifact made me think beyond local solutions and adopt real time, cloud-based data management. One of the most difficult issues I faced was dealing with asynchronous data operations in Kotlin. In loadInventoryFromFirebase(), I defined a callback interface to retrieve inventory items from Firebase and update the RecyclerView after the data was retrieved. This forced me to think reactively where I was waiting for Firebase responses before altering the UI. It definitely helped me better grasp asynchronous programming and how to keep the user interface in sync with real time data changes, but it was challenging at first.

Another problem I faced was refactoring the code to replace SQLite while preserving the user experience. This needed me to have an intricate understanding of how data flows through the app. It also needed me to accurately coordinate between the user interface, data models, and backend functionality. I also had to guarantee that the app could smoothly handle permission checks, login issues, and data sync failures. All of this proved to be challenging, but it taught me a lot as well.

This technique helped me get a better understanding of creating scalable mobile apps that meet real world needs like secure login, fast updates, and ongoing data storage. I now have greater faith in my abilities to design and develop reliable Android apps that function well in real life settings while using modern tools.
